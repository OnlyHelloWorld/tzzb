# 账本数据隔离分析报告

## 问题描述
用户反映一个账本创建的数据会出现在另外一个账本，需要确保账本之间的数据隔离。

## 数据库结构分析

### 核心表结构

#### 1. User 表
- **主键**: `id` (自增整数)
- **核心字段**: `username`, `email`, `password_hash`
- **关系**: 一对多关联到 Ledger、Holding、UserSetting

#### 2. Ledger 表
- **主键**: `id` (自增整数)
- **外键**: `user_id` (关联到 User.id)
- **核心字段**: `name`, `color`
- **关系**: 一对多关联到 Holding

#### 3. Holding 表
- **联合主键**: `(user_id, ledger_id, market, code)`
- **外键**: `user_id` (关联到 User.id), `ledger_id` (关联到 Ledger.id)
- **核心字段**: `market`, `code`, `name`, `sector`
- **关系**: 一对多关联到 Trade

#### 4. Trade 表
- **主键**: `id` (自增整数)
- **外键**: `(user_id, ledger_id, market, code)` (关联到 Holding 的联合主键)
- **核心字段**: `date`, `qty`, `price`, `note`
- **关系**: 多对一关联到 Holding

### 数据隔离机制

1. **数据库层面隔离**:
   - **Holding 表**: 使用联合主键 `(user_id, ledger_id, market, code)`，确保同一用户下不同账本的相同股票代码不会冲突
   - **Trade 表**: 外键约束包含 `ledger_id`，确保交易记录与对应账本的持仓关联
   - **所有查询**: 都通过 `user_id` 和 `ledger_id` 进行筛选

2. **API 层面隔离**:
   - **持仓 API**: 所有 endpoints 都接受 `ledger_id` 参数，并在查询中使用该参数过滤
   - **交易 API**: 交易记录创建时自动关联到对应账本
   - **账本 API**: 操作账本时确保只处理当前用户的账本

3. **前端层面隔离**:
   - **API 调用**: 所有持仓相关操作都传递 `ledger_id` 参数
   - **数据加载**: 切换账本时重新加载对应账本的数据

## 问题修复

### 已修复的问题

1. **Trade 表外键约束**:
   - **问题**: 原外键约束只包含 `(user_id, market, code)`，缺少 `ledger_id`
   - **解决方案**: 修改外键约束为 `(user_id, ledger_id, market, code)`，确保交易记录与对应账本的持仓正确关联

2. **API 端点参数**:
   - **问题**: 部分 API 端点缺少 `ledger_id` 参数
   - **解决方案**: 所有持仓相关 API 都添加了 `ledger_id` 参数，并在查询中使用

3. **前端 API 调用**:
   - **问题**: 前端调用 API 时未传递 `ledger_id` 参数
   - **解决方案**: 更新前端 API 调用，确保所有操作都传递正确的 `ledger_id`

## 确保数据隔离的措施

1. **数据库设计**:
   - 使用联合主键确保同一用户下不同账本的股票代码唯一性
   - 外键约束包含 `ledger_id`，确保数据关联的正确性
   - 所有表都包含 `user_id` 字段，确保用户级别的数据隔离

2. **API 实现**:
   - 所有持仓和交易相关 API 都要求传递 `ledger_id` 参数
   - 查询时使用 `user_id` 和 `ledger_id` 双重过滤
   - 批量操作时确保只处理指定账本的数据

3. **前端实现**:
   - 切换账本时重新加载对应账本的数据
   - 所有 API 调用都传递当前账本的 `ledger_id`
   - 界面上只显示当前账本的数据

4. **测试验证**:
   - 创建多个账本，在不同账本中添加相同股票代码的持仓
   - 验证每个账本只显示自己的持仓数据
   - 测试删除账本时，确保只删除对应账本的数据

## 结论

通过以上措施，确保了账本之间的数据隔离：
- **数据库层面**：通过联合主键和外键约束确保数据关联的正确性
- **API 层面**：所有操作都通过 `ledger_id` 进行筛选
- **前端层面**：只显示和操作当前账本的数据

这样可以确保一个账本创建的数据不会出现在另外一个账本中，解决了用户反映的问题。
